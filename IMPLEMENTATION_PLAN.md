## 1. Продукт (из ТЗ)

- **Кто платит:** собственник бизнеса (50–200 сотрудников, 3–20 точек, высокая текучка,
  стандарты на бумаге не исполняются, перегруз микроменеджментом).
- **Ценность:** собственник передаёт системе всё, что знает о бизнесе — она следит за этим 24/7
  вместо него. Сотрудники получают ответы мгновенно. Стандарты обновляются по отзывам.
- **Три роли (RBAC):**
  - **Собственник** — загружает стандарты, смотрит дашборд, получает алерты, принимает решения.
  - **Управляющий** — контролирует исполнение, видит отчёты по своим точкам.
  - **Сотрудник** — учится через микролёрнинг, спрашивает бота, оставляет обратную связь.
- **Принцип AI-ядра:** строгий RAG. Ответ — только по базе знаний конкретной компании.
  Нет ответа в базе → честно сказать и предложить уточнить у руководителя. Тенанты изолированы.

## 14. Журнал решений (Decision Log)

- **2026-06-11 — План Evergreen перевыпущен по ТЗ v2.** Структура — 6 модулей + Фаза 0 (инфра).
  *Почему:* пришло формальное ТЗ; нужно трекать против него, а не против абстрактного видения.
- **2026-06-11 — Зафиксировано:** харнесс = «мозги» (RAG-чат, классификация, генерация тестов,
  рекомендации). Остальное (БД, auth, интеграции, трекер, push) — обычный бэкенд, строится отдельно.
- **Предложено (требует подтверждения):** vector RAG на **pgvector** (одна БД с Postgres), а не Pinecone —
  дешевле и проще на старте. Подтвердить в Q-сессии.
- *(ранее)* Gemini добавлен как бесплатный провайдер для тестов; снижение стоимости прогона; фронт на Tailwind.

**Спринт 2026-06-11 — бэкенд P0 (`product/`):**
- **Модульный монолит, не два репозитория.** Один репо, два пакета: `composer/` (движок) и
  `product/` (Evergreen). Правило одной стрелки: `product/` → `composer/` только через `product/brain.py`;
  `composer/` про продукт не знает. Разделять репозитории — позже, через `git subtree split`, когда появится второй продукт.
- **БД: PostgreSQL + pgvector** (не Mongo). Та же схема поднимается на SQLite для офлайн-дева и тестов —
  переносимые типы (generic JSON, UUID-строки). *Почему:* реляционная модель + RAG в одной БД, мультитенант по `company_id`.
- **Поиск: косинус в Python поверх JSON-эмбеддингов**, pgvector-индекс — за швом `rag/search.py` на потом.
  *Почему:* нулевая инфра-стоимость сейчас, прод-оптимизация не ломает схему и вызывающий код.
- **Эмбеддер — сменный шов:** GeminiEmbedder (прод) / FakeEmbedder (детерминированный, офлайн/тесты). Выбор через env.
- **Хеш паролей: pbkdf2_sha256, не bcrypt.** *Почему:* связка passlib + bcrypt 4.x ломается («72 bytes»);
  pbkdf2 — чистый Python, без нативных зависимостей, одинаково в деве и проде.
- **Строгий RAG = два гейта:** retrieval (порог близости) + generation (маркер отказа модели). Нет ответа →
  честный отказ + лог `QAGap`. *Почему:* требование ТЗ «без галлюцинаций» + карта пробелов для собственника.
- **Тесты офлайн:** SQLite + FakeEmbedder + фейковый LLM. 29 тестов зелёные, без сети и ключей.
- **Схема при старте через `create_all` (lifespan).** Alembic — следующим шагом (записано в бэклог Фазы 0).

**Спринт 2026-06-12 — прод-фундамент + добивка M1.5/M3 (`product/`):**
- **Alembic вместо «create_all-навсегда».** `alembic/env.py` берёт `DATABASE_URL` и `Base.metadata` из кода;
  на старте Postgres → `upgrade head`, SQLite (дев/тесты) → `create_all`. *Почему:* прод-схему нужно
  версионно мигрировать, а SQLite-тестам версионирование лишнее — поэтому ветка по диалекту.
- **PG-only DDL под гардом диалекта.** pgvector (CREATE EXTENSION/HNSW) и `ADD CONSTRAINT` FK выполняются
  только при `dialect == "postgresql"`; на SQLite их даёт `create_all` из модели. *Почему:* SQLite не умеет
  `ALTER ADD CONSTRAINT`, а гонять одну цепочку миграций на обоих движках удобнее, чем держать две.
- **Хранилище — config-not-code шов (как эмбеддер).** `Storage`-ABC (put/get/exists), бэкенды local/S3,
  выбор по env. Оригиналы под namespaced-ключом `companies/{cid}/documents/{did}/v{n}/{file}`; `source_uri`
  пишется в версию. *Почему:* дев/тесты на диске без инфры, прод — S3/MinIO, без изменения вызывающего кода.
- **pgvector — оптимизация, не смена контракта.** JSON-эмбеддинг остаётся источником правды; на Postgres
  параллельно живёт `vector(N)`-колонка + HNSW, поиск через `<=>`. Нет колонки/не тот размер → молча падаем
  на косинус-в-Python. Проверено вживую на pg16: ранги и скоры идентичны fallback. *Почему:* ускорение при
  росте объёмов без риска для уже работающего пути.
- **EMBEDDING_DIM 1536 → 768.** Под реальную модель Gemini-эмбеддингов (vector-колонка и индекс зависят от размера).
- **M1.5 аудитория = data-filter, не отдельный ACL-слой.** `audience_roles`(JSON)+`point_id`(FK) на документе;
  пустые роли/`point_id=None` = «виден всем». Один хелпер `visible_document_ids` питает и список, и get-по-id,
  и ретривал чата. *Почему:* единая точка правды о видимости → бот не протечёт стандартом чужой роли/точки.
- **M3 треки: оценивание только на сервере.** `submit_step` сам генерит и оценивает тест, хранит лучший балл,
  пересчитывает завершение записи. *Почему:* клиент не должен решать «зачёт/незачёт» — это контролирует бэкенд.
- **Тесты офлайн расширены до 39 зелёных** (+треки, +аудитория). Полная цепочка миграций применяется на чистом SQLite.
