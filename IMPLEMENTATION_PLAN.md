# Evergreen AI — Implementation Plan & Tracker

> **Продукт:** Evergreen AI — AI-операционист для малого и среднего бизнеса с сетью точек
> (кофейни, рестораны, фитнес, фастфуд). Хранит стандарты, обучает сотрудников, собирает
> и анализирует обратную связь, следит за исполнением и шлёт собственнику сигналы.
> **Внутренний движок:** доменно-независимый harness (пакет `composer/`) — «мозги» продукта.
> **Источник требований:** `ТЗ AI-Операционист v2` (MVP, июнь 2026). Этот план — его перевод
> в трекаемую дорожную карту.
>
> **Статусы:** ✅ done · 🟡 in progress · ⬜ todo · 🔒 blocked.
> **Приоритеты ТЗ:** P0 (без этого нет продукта) · P1 · P2.
> Отмечай `[x]` по мере выполнения. Обновляй «Журнал решений» при каждом важном выборе.
>
> ⚠️ Бренд везде, что видит пользователь — **Evergreen**. Имя «Composer»/харнесс/«агенты-воркеры»
> не должны протекать в UI или маркетинг.

## 1. Продукт (из ТЗ)

- **Кто платит:** собственник бизнеса (50–200 сотрудников, 3–20 точек, высокая текучка,
  стандарты на бумаге не исполняются, перегруз микроменеджментом).
- **Ценность:** собственник передаёт системе всё, что знает о бизнесе — она следит за этим 24/7
  вместо него. Сотрудники получают ответы мгновенно. Стандарты обновляются по отзывам.
- **Три роли (RBAC):**
  - **Собственник** — загружает стандарты, смотрит дашборд, получает алерты, принимает решения.
  - **Управляющий** — контролирует исполнение, видит отчёты по своим точкам.
  - **Сотрудник** — учится через микролёрнинг, спрашивает бота, оставляет обратную связь.
- **Принцип AI-ядра:** строгий RAG. Ответ — только по базе знаний конкретной компании.
  Нет ответа в базе → честно сказать и предложить уточнить у руководителя. Тенанты изолированы.

## 4. Фаза 0 — Инфраструктура (фундамент, P0) ⬜

> Без этого модули M1–M6 нельзя довести до прода. Решается ДО или параллельно с M1.

- [ ] ⬜ **0.1 Выбрать и поднять PostgreSQL**
  - *Зачем:* источник правды для документов, пользователей, задач, отзывов, прогресса.
  - *DoD:* схема БД (компании, пользователи, документы, версии, треки, отзывы, задачи), миграции, подключение из API.
- [ ] ⬜ **0.2 Vector RAG (pgvector vs Pinecone)**
  - *Что:* эмбеддинги чанков стандартов, semantic search вместо нынешнего keyword-поиска.
  - *Решение:* pgvector (одна БД, дешевле) — записать в Журнал решений.
  - *DoD:* документ → чанки → эмбеддинги в БД; запрос → top-k релевантных чанков.
- [ ] ⬜ **0.3 Аутентификация + RBAC (3 роли)**
  - *Что:* логин, сессии/JWT, роли Собственник/Управляющий/Сотрудник, скоуп по компании и точке.
  - *Файлы:* `composer/api/app.py` (middleware), новый модуль auth.
  - *DoD:* каждый `/api/*` проверяет роль; сотрудник видит только своё; тенанты изолированы на уровне БД.
- [ ] ⬜ **0.4 Файловое хранилище (S3-совместимое)**
  - *Что:* оригиналы документов, будущие видео-уроки.
  - *DoD:* загрузка/выдача файлов через presigned URL.
- [ ] ⬜ **0.5 Мостик харнесс ↔ БД**
  - *Что:* knowledge-инструменты агента читают из pgvector/БД, а не только из файлов.
  - *Файлы:* `composer/tools/registry.py` (`make_knowledge_tools`).
  - *DoD:* агент-чатбот отвечает по чанкам из БД конкретного тенанта.
- [ ] ⬜ **0.6 PWA-оболочка**
  - *Что:* мобильный доступ без установки (требование ТЗ).
  - *DoD:* приложение ставится как PWA, базовый офлайн-шелл.
