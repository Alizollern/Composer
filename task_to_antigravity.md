## 1. Что это за продукт (контекст, чтобы ты понимал, что строишь)

**Evergreen** — «цифровой двойник руководителя» для сетей малого/среднего бизнеса
(кофейни, рестораны, фитнес, фастфуд). Собственник один раз загружает стандарты компании, а система:

- **отвечает сотрудникам** на вопросы строго по этим стандартам (без выдумок, с цитатой источника);
- **обучает новичков** через короткие уроки + автогенерируемые тесты (онбординг-треки);
- **показывает дыры** — вопросы, на которые в базе нет ответа.

**Три роли (RBAC):**
- **owner (собственник)** — загружает стандарты, видит всё, управляет, смотрит пробелы и прогресс.
- **manager (управляющий)** — управляет стандартами/треками своих точек, видит отчёты.
- **employee (сотрудник)** — учится, проходит тесты, спрашивает бота. Видит только своё.

## 3. Авторизация (как фронт общается с бэком)

Бэкенд — JWT Bearer. Поток:

1. **Регистрация компании** (`POST /api/auth/register-company`) или **вход** (`POST /api/auth/login`)
   возвращают `{ access_token, token_type, role, company_id }`.
2. Токен храни в `localStorage` (ключ, например, `evergreen_token`).
3. Каждый защищённый запрос шлёт заголовок: `Authorization: Bearer <access_token>`.
4. `GET /api/auth/me` возвращает текущего пользователя `{ user_id, company_id, role, point_id }` —
   используй для определения роли и ветвления UI.
5. Если ответ `401` — токен протух/нет → редирект на экран входа.
6. **Роль управляет интерфейсом:**
   - `employee` — видит: чат, свои документы (только своей аудитории), свои онбординг-треки.
   - `manager` / `owner` — всё вышеперечисленное + загрузка/редактирование стандартов, управление аудиторией,
     создание треков, просмотр пробелов и прогресса, создание пользователей.
   - Если эндпоинт вернул `403` — у роли нет прав; прячь такие кнопки заранее по роли.

## 5. Полный контракт API (источник правды — строй UI ровно под это)

Базовый префикс — `/api`. Все тела — JSON, кроме загрузки файла (multipart).
Защищённые эндпоинты требуют `Authorization: Bearer <token>`.
Уровни доступа: **[любой вошедший]**, **[manager+]** (manager или owner), **[owner]**.

### Auth
| Метод | Путь | Доступ | Тело запроса | Ответ |
|---|---|---|---|---|
| POST | `/api/auth/register-company` | публичный | `{company_name, slug, owner_email, owner_password, owner_name}` | `{access_token, token_type, role, company_id}` |
| POST | `/api/auth/login` | публичный | `{slug, email, password}` | `{access_token, token_type, role, company_id}` |
| GET | `/api/auth/me` | любой вошедший | — | `{user_id, company_id, role, point_id}` |
| POST | `/api/users` | manager+ | `{email, password, role, full_name?, point_id?}` | `{id, email, role, full_name, point_id}` |

- `slug` — латиница/цифры/дефис, регэксп `^[a-z0-9][a-z0-9\-]{1,62}$` (валидируй на форме).
- `owner_password`/`password` — минимум 6 символов.
- `role` ∈ `"employee" | "manager" | "owner"`.

### Documents (стандарты)
| Метод | Путь | Доступ | Тело / параметры | Ответ |
|---|---|---|---|---|
| POST | `/api/documents` | manager+ | `{title, content, category?, publish=true, audience_roles?, point_id?}` | `DocumentOut` |
| POST | `/api/documents/upload` | manager+ | multipart: `file`, `category?` | `DocumentOut` |
| GET | `/api/documents?status_filter=` | любой вошедший | query `status_filter?` (`draft`/`published`/`archived`) | `DocumentOut[]` |
| GET | `/api/documents/{id}` | любой вошедший | — | `DocumentOut` (404 если вне аудитории) |
| POST | `/api/documents/{id}/audience` | manager+ | `{audience_roles?, point_id?}` | `DocumentOut` |
| GET | `/api/documents/{id}/original` | любой вошедший | — | файл (download) |
| POST | `/api/documents/{id}/versions` | manager+ | `{content}` | `DocumentOut` |
| POST | `/api/documents/{id}/status` | manager+ | `{status}` | `DocumentOut` |

`DocumentOut = {id, title, category, status, current_version_id?, audience_roles: string[], point_id?}`
- `audience_roles: []` → документ виден всем ролям. `point_id: null` → виден всем точкам.
  Это и есть «теги по роли/точке» (M1.5): в UI редактирования аудитории дай мультиселект ролей + выбор точки,
  с явной подписью «пусто = видно всем».

### Chat (бот по стандартам)
| Метод | Путь | Доступ | Тело | Ответ |
|---|---|---|---|---|
| POST | `/api/chat` | любой вошедший | `{question}` | `{answer, refused: bool, sources: [{document_id, document_title, score}], best_score: number}` |

- Если `refused == true` — бот честно не нашёл ответа. Покажи спокойное сообщение
  («В стандартах пока нет ответа — я зафиксировал вопрос для руководителя»), **без выдумок**.
- Если `refused == false` — покажи `answer` + блок «Источники» с кликабельными `document_title`
  (клик → открыть документ `/api/documents/{document_id}`).

### Gaps (вопросы без ответа — для руководителя)
| Метод | Путь | Доступ | Ответ |
|---|---|---|---|
| GET | `/api/gaps` | manager+ | `[{id, question, best_score, resolved}]` |

### Quiz (тест по документу)
| Метод | Путь | Доступ | Тело | Ответ |
|---|---|---|---|---|
| POST | `/api/documents/{id}/quiz` | manager+ | `{num_questions=5}` | `{document_id, questions: [{question, options: string[], correct_index, source_quote}]}` |
| POST | `/api/quiz/grade` | любой вошедший | `{quiz: Question[], answers: number[]}` | `GradeOut` |

`GradeOut = {total, correct, score, passed, details: [{question, given_index?, correct_index, is_correct, source_quote}]}`

### Tracks (онбординг)
| Метод | Путь | Доступ | Тело | Ответ |
|---|---|---|---|---|
| POST | `/api/tracks` | manager+ | `{title, description?}` | `TrackOut` |
| GET | `/api/tracks` | любой вошедший | — | `TrackOut[]` (сотрудник видит только опубликованные) |
| GET | `/api/tracks/{id}` | любой вошедший | — | `TrackDetailOut` |
| POST | `/api/tracks/{id}/steps` | manager+ | `{document_id, title?, require_quiz=true, pass_score=0.8, num_questions=5}` | `TrackStepOut` |
| POST | `/api/tracks/{id}/status` | manager+ | `{status}` (`draft`/`published`/`archived`) | `TrackOut` |
| POST | `/api/tracks/{id}/enroll` | manager+ | `{user_id}` | `EnrollmentOut` |
| POST | `/api/tracks/{id}/enroll-me` | любой вошедший | — | `EnrollmentOut` |
| GET | `/api/my/enrollments` | любой вошедший | — | `MyEnrollmentOut[]` |
| POST | `/api/enrollments/{eid}/steps/{sid}/submit` | любой вошедший | `{quiz?, answers?}` | `SubmitStepOut` |
| GET | `/api/tracks/{id}/progress` | manager+ | — | `[{enrollment_id, user_id, status, passed_steps, total_steps}]` |

- `TrackOut = {id, title, description, status}`
- `TrackStepOut = {id, document_id, ordinal, title, require_quiz, pass_score, num_questions}`
- `TrackDetailOut = TrackOut + {steps: TrackStepOut[]}`
- `EnrollmentOut = {id, track_id, user_id, status}`
- `MyEnrollmentOut = {enrollment_id, status, track: TrackOut, steps: [TrackStepOut + {progress: {step_id, status, score, attempts}}]}`
- `SubmitStepOut = {step_status, score, attempts, enrollment_status, grade?: GradeOut}`

### Health
`GET /api/health` — пинг бэкенда (для индикатора «связь есть»).

## 6. Экраны, которые нужно построить (это и есть основная работа)

> Все под защитой авторизации, ветвление по роли. Маршруты — пример, можешь уточнить.

### A. Публичные / вход
- **`/` … `/contact`** — маркетинговый сайт (уже есть, сохранить/освежить).
- **`/login`** — вход: поля `slug`, `email`, `password`.
- **`/register`** — регистрация компании (создаёт owner-аккаунт): `company_name`, `slug` (с валидацией),
  `owner_email`, `owner_password`, `owner_name`. После успеха — сохранить токен, редирект в `/app`.

### B. Оболочка приложения `/app/*`
- Левый сайдбар (тёмный `forest`) с навигацией по роли, шапка с названием компании и кнопкой выхода.
- Индикатор связи с бэком (`/api/health`).

### C. Экран «База знаний» `/app/knowledge` — **[manager+ полный, employee только просмотр своего]**
- Список документов (`GET /api/documents`), фильтр по статусу.
- **[manager+]** загрузка: либо вставить текст (`POST /api/documents`), либо файл (`POST /api/documents/upload`,
  поддерживаются txt/md/pdf/docx).
- Карточка документа: заголовок, категория, статус (`draft`/`published`/`archived`), бейджи аудитории.
- **[manager+]** действия: сменить статус, добавить новую версию (текст), **настроить аудиторию**
  (мультиселект ролей + точка; пусто = всем), скачать оригинал.
- Открытие документа — просмотр текста; для employee, если документ вне его аудитории — он его просто не увидит (404).

### D. Экран «Чат» `/app/chat` — **[любой вошедший]**
- Поле вопроса → `POST /api/chat`.
- Ответ: если `refused` — спокойный блок «нет в стандартах»; иначе ответ + блок «Источники»
  (кликабельные названия документов → открыть документ).
- История переписки в рамках сессии (стек сообщений). Можно показать «best_score» как технический индикатор (необязательно).
- Приятный «печатающийся» вид ответа (Framer Motion) — опционально.

### E. Экран «Онбординг (мои треки)» `/app/learning` — **[employee основной потребитель]**
- `GET /api/my/enrollments` — список треков сотрудника с прогрессом по шагам.
- Шаг = документ + (опц.) тест. Поток: «читать документ → пройти тест → следующий шаг».
- Тест: получить вопросы (`POST /api/documents/{id}/quiz` или вопросы приходят в шаге),
  отправить ответы (`POST /api/enrollments/{eid}/steps/{sid}/submit` с `{quiz, answers}`),
  показать результат (`SubmitStepOut.grade`: что верно/неверно + `source_quote`).
- Прогресс-бар трека (passed/total).

### F. Экран «Управление онбордингом» `/app/tracks` — **[manager+]**
- Создать трек, добавить шаги (выбор документа + настройки теста), опубликовать (`status`).
- Записать сотрудника (`enroll`) или дать самозапись.
- Прогресс команды по треку (`GET /api/tracks/{id}/progress`): таблица сотрудник → passed/total → статус.

### G. Экран «Пробелы» `/app/gaps` — **[manager+]**
- `GET /api/gaps` — список вопросов без ответа (`question`, `best_score`, `resolved`).
- Это карта «чего не хватает в стандартах» для руководителя. Сортировка по свежести, фильтр resolved.

### H. Экран «Команда» `/app/team` — **[manager+]**
- Создать пользователя (`POST /api/users`): email, пароль, роль, точка.
- (Список пользователей — если появится эндпоинт; пока минимум — форма создания.)

### Состояния, которые нельзя забыть
- Загрузка (skeleton/spinner), пустые состояния (нет документов/треков/пробелов — дружелюбные заглушки),
  ошибки (показывать текст ошибки от бэка), `401` → разлогин, `403` → прятать действие.

## 8. ДОЛГ БЕЗОПАСНОСТИ N8 — чистка git-истории (отдельная задача)

**Проблема:** в истории git репозитория лежит файл `token` с утёкшим Anthropic-ключом.
Даже если ключ отозван, его нельзя оставлять в истории. Задача — вычистить его из ВСЕЙ истории.

> ⚠️ **Это операция переписывания истории git. Делать аккуратно:**
> - Сначала сделай полную резервную копию репозитория (например `git clone --mirror` в отдельную папку).
> - Согласуй с владельцем: после чистки потребуется **force-push**, и все, у кого есть копии репо,
>   должны будут переклонировать.
> - **Ротацию самого ключа делает владелец вручную** (см. ниже) — это не твоя часть, ты чистишь только историю.

**Шаги (рекомендуемый инструмент — `git filter-repo`):**
1. Убедись, что `token`, `.env`, `*.key` уже в `.gitignore` и `.dockerignore` (если нет — добавь).
2. Удали файл из всей истории:
   ```bash
   # установка: pip install git-filter-repo
   git filter-repo --path token --invert-paths
   ```
   (Альтернатива — BFG Repo-Cleaner: `bfg --delete-files token`.)
3. Проверь, что файла больше нет в истории:
   ```bash
   git log --all --oneline -- token   # должно быть пусто
   ```
4. Принудительно обнови удалёнку (после подтверждения владельца):
   ```bash
   git push origin --force --all
   git push origin --force --tags
   ```
5. Сообщи владельцу: всем нужно переклонировать репозиторий; старые форки/клоны с ключом — удалить.

**Готово, когда:** `git log --all -- token` пуст, удалёнка обновлена, `token`/`.env`/`*.key` в `.gitignore`.
